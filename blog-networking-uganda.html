Networking Fundamentals Every IT Pro in Uganda Must Know

CM
Musamali Christopher
IT Technician & Software Developer, Kampala

Networking knowledge separates the IT technicians who can only handle software complaints from those who can walk into any office environment, understand what is happening end-to-end, and solve problems at the infrastructure level. Whether you are setting up a new office LAN, troubleshooting a connectivity fault, or extending Wi-Fi coverage across a building, these are the concepts you cannot afford to not know.

IP Addressing: The Language of Networks

Every device on a network has an IP address — a numerical label that identifies it. IPv4 addresses are written as four numbers separated by dots, for example 192.168.1.100. The first portion identifies the network; the last portion identifies the specific device on that network. Understanding which part is which requires understanding subnetting.

Private IP ranges — the addresses used on local networks inside offices and homes — fall into three blocks: 10.0.0.0 to 10.255.255.255, 172.16.0.0 to 172.31.255.255, and 192.168.0.0 to 192.168.255.255. When you see a device with an address in one of these ranges, it is on a private network and accesses the internet through a router that translates its private address to a public one (this process is called NAT — Network Address Translation).

Subnetting: Dividing Networks

A subnet mask tells devices which portion of an IP address identifies the network and which portion identifies the host. The mask 255.255.255.0 (written in shorthand as /24) means the first three octets are the network portion — so all devices from 192.168.1.1 to 192.168.1.254 are on the same network, with 192.168.1.255 reserved as the broadcast address. A /24 network gives you 254 usable host addresses.

You do not need to be a subnetting expert to do solid IT support work, but you do need to understand the basics: what a subnet mask is, how to read CIDR notation (/24, /25, /16), and how to determine whether two devices are on the same subnet (if they are not, they cannot communicate directly — they need a router between them).

DHCP and DNS: The Services That Make Networks Usable

DHCP (Dynamic Host Configuration Protocol) is the service that automatically assigns IP addresses to devices when they connect to a network. Your router at home or in the office runs a DHCP server. When a laptop connects to Wi-Fi, the router hands it an IP address, a subnet mask, a default gateway, and a DNS server address — everything the device needs to communicate on the network and reach the internet. When a device cannot get an IP address from DHCP, it self-assigns an address in the 169.254.x.x range — a clear indicator that the DHCP process has failed.

DNS (Domain Name System) translates human-readable domain names into IP addresses. When you type google.com into a browser, a DNS query converts that name into the actual IP address of Google's servers. Common DNS servers include Google's public DNS (8.8.8.8 and 8.8.4.4) and Cloudflare's (1.1.1.1). DNS failure is one of the most frequent causes of "internet not working" complaints — the connection itself may be fine, but if DNS is broken, no domain names will resolve.

Switches and Routers: Different Jobs, Often Confused

A switch connects devices within the same network — it operates at Layer 2 (the data link layer) and forwards traffic based on MAC addresses. Plug 24 computers into a 24-port switch and they can all communicate with each other. A router connects different networks — it operates at Layer 3 (the network layer) and forwards traffic based on IP addresses. Your office router connects the local network (192.168.x.x) to the internet (public IP addresses). Understanding this distinction resolves a surprising amount of diagnostic confusion in the field.

Wi-Fi: The Practical Side

Most IT support in Uganda involves Wi-Fi troubleshooting. Key points: the 2.4GHz band travels further and penetrates walls better but is more congested and slower. The 5GHz band is faster and less congested but has shorter range. For dense office environments, always do a site survey before placing access points — dead zones and interference sources (microwaves, other networks, concrete walls) are not obvious without testing. Use a Wi-Fi analyser app (available free on Android) to see which channels competing networks are using and choose a non-overlapping channel (1, 6, or 11 on 2.4GHz).

Cable Standards and Physical Layer Basics

For any structured cabling work: use Cat6 cable as the minimum standard for new installations. Cat5e is still acceptable for speeds up to 1Gbps, but Cat6 supports 10Gbps at shorter distances and costs little more. Patch cables (the flexible cables that connect devices to wall ports) should be pre-terminated Cat6 or Cat5e. Termination standard matters — use T568B for consistency on both ends of every cable in a building. A cable tester is a worthwhile investment and will save hours of fault-finding on new installations.

The Troubleshooting Framework

When a user reports "no internet", work from the bottom of the network stack upward: first confirm the physical connection (is the cable plugged in? Is there a link light on the port?). Then confirm the IP configuration (run ipconfig on Windows — does the device have a valid IP address, or a 169.254 APIPA address?). Then confirm gateway reachability (ping the default gateway — if this fails, the problem is between the device and the router). Then confirm internet reachability (ping 8.8.8.8 — if this works but websites do not load, the problem is DNS). Then confirm DNS (ping google.com by name — if this fails, check DNS settings). This sequence eliminates whole layers of possibility with each step and gets you to the real fault quickly.

Keep Learning

Networking is a deep discipline and this article is a starting map, not a complete reference. The CompTIA Network+ and Cisco CCNA certifications both provide structured and comprehensive networking education — either is worth pursuing if you are serious about IT infrastructure work. In the meantime, the best learning happens on real networks: set up a home lab, configure a switch, build a small VLAN, troubleshoot a real fault. Theory without practice does not stick.
